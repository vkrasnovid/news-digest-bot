# Fix 9 HIGH/MEDIUM QA bugs in clicker game index.html

**Date:** 2026-03-20 10:00
**Files:** index.html
**Severity:** high (3 HIGH, 6 MEDIUM)

## Problem

QA report identified 3 HIGH and 6 MEDIUM bugs in the single-file SPA:
- H-1: Mobile shop cards made affordable by updateUI() missing `touchend` listener
- H-2: Auto-clicker accumulates fractional coins causing display lag
- H-3: `tapCount` not reset on Prestige, breaking Mega Tap cadence
- M-1: Double tap event (touchstart + click) can fire handleTap twice on Android
- M-2: megaTap upgrade level has no effect on bonus magnitude (always ×100)
- M-3: `shareScore` uses `switchInlineQuery` which requires an inline bot registration
- M-4: `goldRushActive` / `goldRushEndTime` not persisted to localStorage
- M-5: Multiplier upgrade card shows "×1.0" at level 0, misleading players
- M-6: No numeric validation in `loadState()` — corrupted save propagates NaN through state

## Root Cause

- H-1: `updateUI()` has a lightweight affordability path that adds only `click`, not the `touchend` that `renderShop()` adds; inconsistency between two code paths.
- H-2: `perSecond * prestigeMultiplier` can produce floats; no floor applied before accumulation.
- H-3: `tapCount` is module-level, not part of `state`, so `resetState()` never touched it.
- M-1: `touchstart` + `click` both registered; no guard to suppress the synthetic click that follows a touch event on some webviews.
- M-2: `getUpgradeEffect('megaTap')` computes level-based value but `handleTap()` hardcodes `×100` and ignores it.
- M-3: `tg.switchInlineQuery` is an inline-mode API; game bots typically are not inline bots.
- M-4: `goldRushEndTime` was a module-level runtime variable, never written into `state` before JSON serialisation.
- M-5: Formula `1 + 0.5 * 0 = 1.0` shown verbatim with no indication it's the pre-purchase value.
- M-6: Deep merge `{ ...def, ...saved }` trusts all saved values without numeric coercion.

## Solution

- H-1: Added `touchend` listener alongside `click` in the `updateUI()` disabled→enabled branch.
- H-2: Wrapped auto-clicker gain in `Math.floor()` before adding to state.
- H-3: Added `tapCount = 0` and `goldRushEndTime = 0` inside `resetState()`.
- M-1: Introduced `_lastTouchTime` flag; `click` handler skips if fired within 350 ms of a `touchstart`.
- M-2: `gain *= 100 * Math.max(1, getUpgradeEffect('megaTap'))` so each level scales the bonus.
- M-3: Replaced `tg.switchInlineQuery` with `tg.openTelegramLink` using a standard share URL.
- M-4: `saveState()` now writes `state.goldRushEndTime = goldRushEndTime` before serialising. `activateGoldRush()` accepts an optional `duration` param. `DOMContentLoaded` restores an active Gold Rush if `state.goldRushEndTime > Date.now()`.
- M-5: At level 0, multiplier card shows "×1.5 после покупки" instead of computing the meaningless "×1.0".
- M-6: After merge in `loadState()`, all numeric fields are clamped with `Math.max(0, Number(x) || 0)`.

## Prevention

- When two code paths both attach event listeners (renderShop vs updateUI), keep them in sync or extract a shared helper.
- Always `Math.floor` / `Math.round` passive income ticks before accumulation.
- Module-level game variables that must reset with game state should be inside `resetState()` — list them explicitly.
- For touch + click dual registration, always add a timestamp or flag guard.
- Upgrade effect functions should be wired end-to-end in the mechanic that uses them; grep for the effect function call site when adding a new upgrade.
- Prefer `tg.openTelegramLink` for sharing; `switchInlineQuery` is only for inline bots.
- Any runtime state (timers, active flags) that affects gameplay must be written into the persisted state object before `JSON.stringify`.
- After loading from untrusted storage, validate and clamp all numeric fields.
